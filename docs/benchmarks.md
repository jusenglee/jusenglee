# Benchmarks & Public Metrics

이 문서는 공개 가능한 문서와 테스트 결과를 기준으로, 포트폴리오에서 인용 가능한 수치만 정리한 문서입니다.  
내부 운영 환경의 민감 정보나 비공개 지표는 포함하지 않았습니다.

---

## 1. NTIS RAG + Triton 통합 시험

초기 통합 시험에서는 Qdrant 기반 RAG와 Triton LLM을 결합한 파이프라인의 기본 동작과 응답 지연을 검증했습니다.

| 항목 | 값 | 비고 |
|---|---:|---|
| Qdrant 검색 평균 응답 | 0.4초 | 초기 통합 시험 기준 |
| Triton 모델 응답 평균 | 4.4초 | 초기 통합 시험 기준 |
| End-to-End 평균 처리시간 | 5.3초 / 질의 | 검색 + 생성 포함 |
| 키워드 매칭 정확도 | 85% 이상 | RapidFuzz 기반 |
| 데이터 규모 | 약 90만 건 | 공개 영문 논문 풀텍스트 기반 시험 데이터 |

### 메모
- 키워드 확장, 재랭킹, 중복 제거, 토큰 클램핑이 포함된 흐름으로 시험했습니다.
- 본 수치는 **실제 NTIS 운영 환경**이 아니라, **초기 통합 검증 시험 환경** 기준입니다.

→ [NTIS AI 챗봇 케이스 스터디](../case-studies/ntis-rag-chatbot.md)

## 2. 임베딩 모델 비교 평가 (100,000 문서 기준)

동일 데이터셋·하드웨어 환경에서 `multilingual-e5-large`와 `bge-m3`를 비교했습니다.

| 항목 | multilingual-e5-large | bge-m3 |
|---|---:|---:|
| 전체 처리 시간 | 약 24.4분 | 약 28.9분 |
| Total wall time | 1450.73s | 1734.72s |
| Embedding time | 1301.44s | 1588.53s |
| 처리량 | 76.8 docs/s | 63.0 docs/s |

### 해석
- 속도와 운영 안정성, 다국어 커버리지를 함께 고려할 때 `multilingual-e5-large`가 기본 임베딩으로 적합하다고 판단했습니다.
- `bge-m3`는 구조 특성상 기술 상세 문서나 긴 문서 처리 측면의 강점이 있었지만, 운영 기본값 관점에서는 e5-large 쪽이 더 실용적이었습니다.

→ [Oracle → Embedding → Vector DB Pipeline 케이스 스터디](../case-studies/oracle-to-qdrant-pipeline.md)

## 3. 모두의 R&D 질의의도 분석 시스템

이 프로젝트는 공개 가능한 범위에서 **정형 출력 일관성**과 **운영형 서빙 구조**를 중심으로 정리했습니다.

| 항목 | 공개 가능한 근거 |
|---|---|
| 입력 데이터 | IRIS 개선의견 테스트 시트 다건 입력 |
| 출력 구조 | 교정 / 구조화 요약 / 주제 분류 |
| 분류 체계 | T1~T5 토픽 축 |
| 운영 환경 | 온프레미스 GPU + Triton 기반 서빙 |
| 정량 성능 | 공개 가능한 일관 수치 없음 |

### 메모
- 테스트 시트에는 약 36건의 의견 데이터에 대해 교정·요약·분류 결과가 채워져 있음을 확인했습니다.
- 정량 성능보다는 **실무형 구조 출력**과 **운영 가능한 서빙 구조**가 핵심이었습니다.

→ [모두의 R&D 케이스 스터디](../case-studies/everyones-rnd-intent-classification.md)

## 5. 해석 원칙

- 수치는 **문서로 확인 가능한 공개 범위**만 사용합니다.
- 시험 환경 수치를 운영 환경 수치처럼 표현하지 않습니다.
- 정량 수치가 부족한 프로젝트는 **출력 구조, 운영 구조, 제품화 범위**로 설명합니다.
